BST 有序表

经典BST中无重复值

BST的增删查

增和查操作相似

删除的三种情况:

  1. 无left无right, 直接删

  2. 有left无right 或 有right无left(情况相同, 子树替他的位置),

  3. 有left有right, 找后继节点替他的位置(右树的最左节点), 后继节点如果有right child, 把right child交给后继节点的parent;

    用后继节点接替时要考虑后继节点是要删除节点的right child, 这种情况下, 后继节点无需交出自己的right child

裸的BST的问题及解决:

  • Question: 输入数据状况决定性能: 数据问题导致树的不平衡, 复杂度无法保持在O(logN)
  • Solution: BST引入左旋和右旋构成平衡BST, 具备自平衡能力, 维持平衡的代价不超过O(logN)

常见平衡BST(self-balancing binary search tree)

AVL(Adelson-Velskii and Landis)

任何节点, 左右树高度差 <= 1

因为每一步都会检查平衡性, 所以左右高度差在调整前最大就是2

四种违规类型: RL, RR, LL, LR, 如果中了两种, 一种操作即可调整到平衡

每种违规情况的调整代价都是O(1), 即便这条链往上每个节点都调整, 代价最多是log(N)

Size-Balanced Tree

叔节点的个数不少于侄子节点的个数

Size-Balance Tree Implementation

Red Black Tree

TreeMap(SortedMap接口) 底层是红黑树

TreeMap增删改查: O(logN) <===> HashMap的增删改查: O(1)

  1. 节点非黑即红

  2. root、leaf必须是黑

  3. 红的child必须是黑

  4. 任何一条链从头到叶黑色节点一样多

  5. 最长和最短的链不会到达两倍以上

插入节点: 5种违规情况

删除节点: 8种违规情况

相比于AVL树, SB树和红黑树的平衡性相对模糊

扰动: SB树和红黑树变化的不像AVL那么频繁, 这种频繁扰动会对磁盘I/O产生影响

磁盘中的树结构: B树, B+树(查询效率比SB树和红黑树低, 但是不用频繁的写硬盘)

Skip List 跳表

也可以做有序表, 复杂度也是O(logN)

构成:

  • 内部的节点是Node, 有多条往外指的指针 List

  • 最左侧节点的key是null, 且能扩充向外指的指针个数; 用ArrayList里面加1个null表示有一个指针, 多加几个null就表示指针扩充了

  • 新加Node至少有一个指针, 往外扩充的指针由Math.Random决定

  • 每次查询都是从最高层指针往下, 类似建立索引, 高层节点走过一个Node, 底层会走过很多个Node
